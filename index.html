Hypercore is a distributed append-only log

Hypercore comes with a secure transport protocol, making it easy to build fast and scalable peer-to-peer applications. Think lightweight blockchain crossed with BitTorrent.

Hypercore currently serves as the foundation for a diverse range of P2P applications, including chat apps, filesystems, databases, and even a browser for the distributed internet. And the whole thing's MIT-licensed, so you can use it as you like!

If you are interested in trying it out we suggest you go to the Github repository of the Node.js implementation.

To get in touch with us, you can join our Discord or shoot us an email at hello@hypercore-protocol.org

Append-only logs?

If you are not familiar with append-only logs, they are basically just lists you only can append. If you think about it in terms of normal array operations, it is a log where you can only call get(index), push(data) and retrieve the log length, but where you can never overwrite old entries.

Hypercore allows you to quickly distribute these kind of logs in a peer to peer fashion. Each peer can choose to download only the section of the log they are interested in in a random-access fashion, without having to download everything from the beginning.

Using append only logs, Hypercore can easily digest and share between peers which part of a log a peer has through the use of compressed bitfields. This, among other things, helps make the replication protocol light and efficient.

Secured by Merkle trees and cryptography

Hypercore allows you to quickly distribute logs in a peer to peer fashion. It builds on a model where the log contents is verified by a merkle tree using the BLAKE2b-256 hash function. Through this merkle tree, peers can choose to only download the parts of the log they are interested in, and redistribute these parts to others.

Merkle trees are widely used by many distributed systems in various forms. Since Merkle trees are normally only used to verify static content, i.e. if the content changes the root of the Merkle tree changes, Hypercore utilises asymmetric cryptography to sign the root of the Merkle tree as data is appended to it.

As data is appended to the log, the Merkle tree grows. As it grows the root of the tree is signed with EdDSA. Only the owner of the private key can grow the log. Using multiple Hypercores, multi user data structures can be constructed. Hypercore's main purpose is to be the building block of other things.

Hyperdrive is a P2P filesystem

Hyperdrive is designed to help you share files quickly and safely, directly from your computer.

Hyperdrive is built using Hypercores. Using an append-only index you can find only the files you are interested from a large drive in milliseconds - all P2P.

By default, readers only download the portions of files they need, on demand. You can stream media from friends without jumping through hoops! Seeking is snappy and there's no buffering.

If you want to learn more about the code you can see the Node.js implementation in the Github repository

If you want to download (this site is even available, using the link at the bottom!) or share filesystems, we suggest you read on and try out the daemon.

The Hyperdrive Daemon

The cross-platform Hyperdrive daemon deals with all the storage and networking details for you. The daemon's CLI lets you create, share, and inspect Hyperdrives in your library, and its gRPC API provides full programmatic control over daemon-managed drives.

You can install the daemon today using the NPM package manager.

npm install -g hyperdrive-daemon

If you're on Mac or Linux, the daemon also offers FUSE support, meaning your drives will appear as normal folders on your computer -- moving files in and out of Hyperdrive is a breeze.

As with our other modules and data structures, you can read more about the daemon in Hyperdrive Daemon Github repository.

How does Hyperdrive work?

Hyperdrive indexes filenames into a Hypercore. To avoid having to scan through this entire Hypercore when you want to find a specific file or folder, filenames are indexed using an append-only hash trie, we call Hypertrie. The hash trie basically functions as a fast append-only key value store with listable folders.

If you are looking for a simple database abstraction on top of Hypercore you can also use Hypertries standalone, outside of Hyperdrive. In you want to learn more about how to use it, we suggest you look at the Hypertrie repository

Since it builds on top of the append-only log, it inherits the same guarantees of every change being versioned by default, making it easy to see historical changes and prevent accidental data loss.

In addition to the Hypertrie, which we refer to as the metadata log, Hyperdrive uses another Hypercore to store the binary file content of each file you insert. Using this dual log design, it makes it easy to replicate or watch only the metadata if that is what you are interested in.

Each entry in the Hypertrie links out to this content log to signal where the binary data starts and ends for this specific. Additionally the entry contains all the normal POSIX data you'd be interested in, such as modification time, created time, file modes, etc.

Mounts

For better composability and collaboration, an entry can also link to a completely different Hyperdrive or Hypercore. We call this feature mounts, and even though it's perceivably simple, it is one of the more powerful features of Hyperdrive for collaboration.

Internally we have been using Hyperdrive mounts for a concept we call "groupware", where each user mounts their own drive inside a single shared one. This allows you to construct a Dropbox like experience without having to worry about conflicts and likewise collaboration problems.

We are actively exploring even more interesting ways of indexing files and collaborating with them. If you are interested in collobarating with us always feel free to to open an issue on Github, reach out at hello@hypercore-protocol.org or join our Discord.

Hyperswarm is a DHT for the home

Hyperswarm combines a Kademlia-based distributed hash table (DHT) for global discovery with MDNS to discover peers on local networks.

Each component of Hyperswarm is built with home networks in mind, meaning that any DHT peer can you help you establish a connection to another peer they know about. This technique is called "UDP holepunching", and is what allows Hyperswarm to make P2P connections without having to worry too much about firewalls.

Hyperswarm is part of the Hypercore protocol but can be used standalone as well. You can learn more on the Hyperswarm Github organisation and in the main Node.js implementation repository

A Kademlia DHT

Like most other P2P projects, Hyperswarm uses a DHT to discover peers on the internet. In the Hypercore protocol, a hash of the Hypercore's public key is used as the topic under which peers share their IP and port to discover each other.

Hypercore comes with a capability system where each peer has to verify they know the public key before they start sharing data, so you have to know the public key in order to download data from peers (even if you’re watching the DHT).

The DHT used in Hyperswarm is based on Kademlia. It is a flexible DHT where each request/response sent is a simple extensible Protocol Buffers message. You can read much more about Kademlia in the Kademlia paper, but it is a UDP-based DHT which is used by many projects, including BitTorrent.

Distributed Hash Tables scale well in general, but can also be notoriously hard to work with. Hyperswarm employs a series of heuristics in order to answer queries and garbage collected stale data as fast as possible.

You should note that DHTs in general expose routing information such as your IP and port to help route other peers to you. Better privacy preserving DHTs are an active research area by many participants in the P2P ecosystem.

Distributed Holepunching

P2P networks mean connecting to peoples computers where-ever they are. This is often a challenge due to firewalls which reject incoming connections and due to NATs which mask your IP. To help “break through” firewalls, a technique called UDP holepunching is used.

Traditionally UDP holepunching requires the use of centralised servers that are preknown by each peer in a network.

Hyperswarm expands on holepunching by making it a first class feature of its DHT. This means that any peer in the DHT can help you holepunch to any peer that the DHT node knows about.

After finding the relevant peers, Hyperswarm utilises the UTP transport protocol, to make reliable connection streams between peers. For non-holepunching cases, TCP is used as well.

Hypercore utilises the Noise protocol framework to build end-to-end encrypted connections on top of the UTP/TCP streams to other peers provided by Hyperswarm.
